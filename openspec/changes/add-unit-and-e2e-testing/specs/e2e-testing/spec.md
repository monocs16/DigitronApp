## ADDED Requirements

### Requirement: Playwright is configured for E2E testing against local Supabase

The system SHALL have a `playwright.config.ts` pointing to `http://localhost:5173` as `baseURL`,
with two named projects: `admin` and `technician`, each using a pre-saved `storageState` for
authentication. A global setup script SHALL generate these storage state files by logging in via
the app's login page. A `test:e2e` npm script SHALL invoke Playwright in headless mode.

#### Scenario: E2E suite runs with a single command

- **WHEN** `bun run test:e2e` is executed with the local Supabase stack and dev server running
- **THEN** all E2E tests complete and the command exits with code 0

#### Scenario: Unauthenticated access redirects to login

- **WHEN** Playwright navigates to `/orders` without a stored auth session
- **THEN** the browser is redirected to the login page

---

### Requirement: Admin can complete the full service-order happy path

The system SHALL have an E2E test that drives the `admin` authenticated context through all
mandatory stages of a new service order: intake → evaluation → budget → customer_decision →
repair → payment → delivered → closed.

#### Scenario: Admin creates a new order

- **WHEN** admin navigates to `/orders/new`, fills in client, equipment, problem, and submits
- **THEN** the browser redirects to the new order's detail page and the stage badge shows `intake`

#### Scenario: Admin advances order to evaluation

- **WHEN** admin clicks the "Move to Evaluation" action on an `intake` order
- **THEN** the stage badge updates to `evaluation`

#### Scenario: Admin advances order to budget

- **WHEN** admin is on an `evaluation` order and clicks "Submit to Budget"
- **THEN** the stage badge updates to `budget`

#### Scenario: Admin approves the budget

- **WHEN** admin fills in labor cost and clicks "Approve" on a `budget` order
- **THEN** the stage transitions to `customer_decision` with decision set to `approved`

#### Scenario: Admin starts repair

- **WHEN** admin clicks "Start Repair" on a `customer_decision` order with approved budget
- **THEN** the stage badge updates to `repair`

#### Scenario: Admin registers payment and marks delivered

- **WHEN** admin fills in payment amount and method on a `repair` order and confirms payment
- **THEN** the stage transitions to `payment` and then `delivered`

#### Scenario: Admin closes the order

- **WHEN** admin clicks "Close Order" on a `delivered` order
- **THEN** the stage badge updates to `closed` and no further actions are available

---

### Requirement: Technician sees only assigned orders

The system SHALL have an E2E test that verifies role-gated visibility: a technician user can view
the order detail for orders assigned to them but cannot access orders assigned to other technicians
or advance stages that require admin role.

#### Scenario: Technician can view assigned order

- **WHEN** technician navigates to the detail page of an order assigned to their user
- **THEN** the page renders the order detail without an auth error

#### Scenario: Technician cannot advance intake stage

- **WHEN** technician is on an `intake` order detail page
- **THEN** the "Move to Evaluation" action is not rendered (admin-only transition)

#### Scenario: Technician can submit evaluation on assigned order

- **WHEN** technician fills in the diagnosis field on an `evaluation` order assigned to them and submits
- **THEN** the evaluation is saved and the stage advances to `budget`

---

### Requirement: Auth storage state is generated by global setup

The system SHALL have a `e2e/global-setup.ts` that launches a headless browser, navigates to the
login page, signs in as each test user (admin and technician), and saves the resulting browser
storage state to `e2e/fixtures/admin-state.json` and `e2e/fixtures/technician-state.json`.

#### Scenario: Storage state files are created before tests run

- **WHEN** Playwright runs with the configured `globalSetup`
- **THEN** `e2e/fixtures/admin-state.json` and `e2e/fixtures/technician-state.json` exist before
  any test starts

#### Scenario: Stored session is valid for the full test run

- **WHEN** a Playwright test uses the `admin` project
- **THEN** the test starts already authenticated without visiting the login page
